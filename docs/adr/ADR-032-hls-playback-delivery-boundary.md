# ADR-032: HLS playback delivery boundary

- **Status:** Accepted
- **Date:** 2026-06-18
- **Deciders:** DubBridge platform team
- **Closes:** X25 / S-125 planning gap

## Context

S-120 prepares media by probing source artifacts and producing a canonical HLS
package: one or more `.m3u8` playlists plus segment files stored behind
`StorageAdapter`. That package is useful only if later product surfaces can request
it safely.

The missing decision is where playback delivery belongs. If downstream slices read
HLS objects directly from storage, they would learn storage-key layout, bypass
backend authorization, and risk exposing prepared media before the relevant review
or publication gates are satisfied. If each consumer invents its own playback
route, the platform will duplicate manifest rewriting, segment access, token
expiry, tracing, and fail-closed readiness checks.

DubBridge therefore needs an explicit playback-delivery boundary between prepared
media and user-facing playback.

## Decision

### S-125 owns HLS playback delivery

Add a dedicated roadmap slice, `S-125` - HLS playback delivery. It is planned after
`S-120` because it depends on prepared HLS artifacts and before review/publication
runtime consumers because those consumers need one stable media-delivery contract.

`S-125` delivers the server-side playback boundary only. It does not create a
public website, revive the retired authenticated web console, or define consumer
player UX. Product UI work remains with the consuming slices and with any future
public-player decision.

### Prepared HLS is never exposed as raw object-storage keys

Clients must not construct object-store paths or fetch prepared HLS files directly
from MinIO/S3 keys. Playback access goes through a backend-owned grant boundary
that validates:

- the prepared-media readiness state from S-120;
- caller authentication and org/project authorization for authenticated review
  playback;
- publication visibility once S-180 exposes audience-facing delivery; and
- expiry/scope of any generated playback token or signed URL.

The implementation may use short-lived signed object-store URLs, backend proxying,
or CDN-backed signed URLs, but the API contract is the same: the backend grants
access, and storage keys remain implementation detail.

### Manifest and segment URLs are generated by the playback boundary

HLS manifests must be returned in a form the client can follow without knowing
storage layout. The playback boundary either rewrites manifests to routed segment
URLs or issues manifests whose segment references are already scoped, expiring,
and authorized.

Segment access must remain bounded by the same playback grant, but the boundary may
enforce that either by re-checking the grant on every segment request or by issuing
short-lived, scoped, expiring segment references when the manifest is generated. A
valid manifest alone is not a durable permission to fetch media forever.

### Playback is readiness-gated and publication-gated

Playback fails closed when the prepared HLS package is missing, malformed, still
processing, or marked failed by S-120. Review-time playback is additionally gated
by the authenticated reviewer/workspace rules from S-100/S-160/S-170.
Audience-facing playback is gated by the S-180 publication runtime and the
fail-closed publication decision from ADR-030.

### Observability is split between durable grants and high-volume segment traffic

Creating or refusing a playback grant is governance-relevant and must produce
durable, traceable observability in line with ADR-018. Individual segment requests
are high-volume delivery traffic; they must carry trace/correlation data and
metrics, but they do not require one durable audit row per segment unless a later
policy requires it.

## Consequences

**Positive**

- HLS playback cannot be forgotten as an implicit S-180 detail; it has its own
  roadmap slice and ADR.
- Storage keys remain private implementation details owned by `crates/storage`.
- Review and publication runtime share one playback contract instead of inventing
  separate media-serving paths.
- The mobile-only authenticated product-surface decision from ADR-029 remains
  intact because S-125 is a delivery API boundary, not a new authenticated web UI.

**Negative / trade-offs**

- `S-125` adds one more slice between preparation and full review/publication
  runtime.
- Manifest rewriting or signed segment URL generation adds complexity beyond
  simply returning object-store URLs.
- CDN/public delivery remains intentionally undecided until S-180 or a later
  public-distribution slice defines the audience-facing product requirement.

## Alternatives considered

- **Put all HLS serving in S-120** - rejected: S-120 prepares artifacts and should
  not also own client delivery, authorization grants, or publication visibility.
- **Hide playback inside S-180 only** - rejected: review runtime may need the same
  playback capability before publication, and hiding it inside S-180 makes it easy
  to miss.
- **Expose raw S3/MinIO keys to clients** - rejected: leaks storage layout and turns
  object-store access into an accidental authorization boundary.
- **Create a public web/player slice now** - rejected: ADR-029 says future public
  web/player work is a separate product decision. S-125 defines the backend
  delivery boundary that such a product could consume later.
