# Plan: S-125 ŌĆö HLS Playback Delivery

> **Status:** Done 2026-06-22. Authored after the roadmap review that created the
> `S-125` phase and ADR-032. Grant issuance (`T1` through `T4d`), manifest delivery
> (`T5a`), short-lived segment references (`T5b`), and ADR/docs propagation (`T5c`)
> are implemented and verified. ADR-032 is now `Accepted`.
> **Roadmap phase:** `S-125` ŌĆö HLS playback delivery (authorized `.m3u8` + segment serving).
> **Tasks ledger:** `docs/tasks/s-125-hls-playback-delivery.md`.
> **Governing ADR:** `docs/adr/ADR-032-hls-playback-delivery-boundary.md` (Accepted).

## Purpose

`S-120` turns a source artifact into prepared media: durable probe metadata plus a
canonical HLS package (`.m3u8` playlists + segments) stored behind `StorageAdapter`.
That package is storage-backed lineage, not a client contract. Nothing in the
platform can yet hand a prepared manifest to a consumer safely.

`S-125` closes that gap by adding the server-side playback-delivery boundary between
prepared media and any user-facing playback. It lets review-time and (later)
publication-time consumers request prepared HLS through a backend-owned grant that
enforces readiness, authorization, scope, and expiry fail-closed ŌĆö without ever
exposing raw object-store keys.

Without this slice, every downstream runtime consumer (`S-170` review runtime,
`S-180` publication runtime) would reinvent manifest rewriting, segment access,
token expiry, tracing, and readiness gating, and would risk leaking storage layout
or serving prepared media before the relevant gate is satisfied (ADR-032 Context).

## Objective

Deliver a fail-closed HLS playback-delivery boundary that:

- issues backend-owned, scoped, expiring **playback grants** for a prepared asset
  instead of exposing object-store keys;
- serves `.m3u8` manifests rewritten so the client follows short-lived, scoped,
  expiring segment references without knowing storage layout;
- binds segment access to the same grant by minting short-lived segment references
  at manifest-delivery time, so a manifest is not a durable permission to fetch
  media forever;
- gates playback on `S-120` prepared-media readiness and, for review playback, on
  the `S-100`/`S-160` authenticated workspace/reviewer rules;
- reserves audience-facing playback for the `S-180` publication runtime and ADR-030's
  fail-closed approval rule (this slice provides the boundary, not the audience policy);
- emits durable, traceable observability for grant creation/refusal (ADR-018) while
  treating high-volume segment traffic as metric/trace-only delivery.

## Scope

### Included

- A **playback grant** contract: a backend-issued, scoped, expiring authorization to
  play one prepared asset's HLS package, persisted as the system of record.
- A manifest-delivery path that returns a `.m3u8` whose segment references are
  scoped/expiring, never raw object-store keys.
- A short-lived segment-reference mechanism bound to the same grant (signed URL or
  equivalent scoped reference), rather than a durable direct object-store path.
- Readiness gating against the `S-120` `PreparationStatus` (`Ready` only) and the
  derived `HlsManifest`/`HlsSegment` lineage rows.
- Review-time authorization integration with the `S-100`/`S-160` org/project/reviewer
  policy already enforced at the Axum boundary (ADR-023, ADR-027).
- Split observability: durable audit + trace for grant decisions; trace/metrics for
  segment traffic (ADR-018, ADR-032).
- API contract surface for `apps/api` (grant issuance, manifest fetch, segment fetch)
  and the storage-key abstraction it depends on.

### Excluded

- A public website, a revived authenticated web console, or any consumer player UX
  (ADR-029, ADR-032). `S-125` is a delivery **API boundary**, not a UI.
- Audience-facing publication visibility policy and CDN/public distribution ŌĆö owned by
  `S-180` and a later public-distribution decision (ADR-032 ┬¦Consequences).
- Producing the HLS package itself (owned by `S-120`) and any re-transcode, alternate
  rendition ladder, DRM, or per-segment encryption beyond what `S-120` emits.
- The review/publication runtime behavior (`S-170`/`S-180`); this slice supplies the
  shared playback contract those slices consume, not their workflows.

## Governing constraints

- **ADR-032:** prepared HLS is never exposed as raw object-store keys; the backend
  grants access; manifests/segments are generated by the playback boundary; playback
  is readiness-gated and publication-gated; observability is split between durable
  grants and high-volume segment traffic.
- **ADR-006:** PostgreSQL remains authoritative for grant metadata; binaries remain
  immutable object-store artifacts referenced by key + checksum.
- **ADR-008 (spirit):** playback is a fail-closed precondition ŌĆö missing/processing/
  failed preparation, an expired or unauthorized grant, or an unknown stored state
  must deny, never degrade.
- **ADR-018:** grant create/refuse is governance-relevant and needs durable, correlated
  observability; segment traffic carries trace/correlation + metrics.
- **ADR-023 / ADR-027:** caller identity is verified at the Axum boundary and review
  playback is org/project-scoped via the existing membership guard; this slice does not
  weaken or fork that boundary.
- **ADR-030:** audience-facing playback honors the latest fail-closed approved review
  decision; `S-125` exposes the hook, `S-180` enforces the audience policy.
- **Roadmap X25 / X12:** deliver HLS playback delivery without exposing raw keys, and
  preserve lineage/quality-gate transitions for the derived artifacts being served.

## Design decisions

### D1 ŌĆö The playback grant is the system of record, not the manifest

A `.m3u8` is a transient view. Authorization, scope, and expiry live in a durable
**playback grant** row (PostgreSQL, ADR-006). A manifest is rendered from a grant;
holding a manifest never extends access forever because the segment references emitted
into that manifest are themselves short-lived and scoped to that grant window.

### D2 ŌĆö Manifest rewriting is a pure, IO-free transform

`crates/playback` owns a pure function that takes the prepared `.m3u8` plus a grant
context and returns a rewritten manifest whose segment references are scoped/expiring.
It performs no storage IO and no network calls, mirroring the
`crates/media` pure-builder seam, so manifest rewriting is fully unit-testable. The
executor layer in `apps/api` performs the storage reads and short-lived reference
generation.

### D3 ŌĆö Storage keys never leave `crates/storage`

Following the `S-080`/`S-120` key-ownership rule, API/playback code must not construct
prepared-HLS object keys. `crates/storage` exposes the canonical read/reference helper;
the playback boundary asks storage for a scoped reference or streamed body and never
emits a raw MinIO/S3 path to a client (ADR-032).

### D4 ŌĆö Playback fails closed on readiness and authorization

Grant issuance requires `PreparationStatus::Ready` for the asset and a resolvable
`HlsManifest` lineage row from `S-120`. Missing, `Pending`/`InProgress`, `Failed`, or
malformed preparation denies issuance. An unknown stored grant status decodes to an
explicit `UnknownStoredValue`-style denial, never an implicit allow (ADR-008 spirit).

### D5 ŌĆö Authorization is layered, not forked

Review-time grant issuance reuses the existing verified principal (ADR-023) and the
`S-100`/`S-160` org/project membership guard (ADR-027). `S-125` adds a playback-scope
check on top; it does not introduce a parallel auth path or weaken the Axum boundary.
The audience-facing path is intentionally a hook the `S-180` publication runtime fills
(ADR-030); `S-125` does not implement audience visibility itself.

### D6 ŌĆö Observability is split by volume and governance weight

Grant create/refuse produces a durable audit row + correlated trace (ADR-018). Segment
delivery is high-volume traffic: v1 records metrics/access evidence at the signed-
reference or storage boundary and does not write one durable audit row per segment
unless a later policy requires it (ADR-032).

### D7 ŌĆö Delivery mechanism is pluggable behind one contract

ADR-032 permits short-lived signed object-store URLs, backend proxying, or CDN-signed
URLs. `S-125` fixes the **API contract** (grant ŌåÆ manifest ŌåÆ short-lived segment
reference) and selects one v1 mechanism in the implementation task; the contract does
not change if the mechanism is later swapped. **Chosen v1 direction:** validate the
grant strongly at manifest delivery, then emit short-lived scoped segment references
instead of re-authorizing every segment request through the API.

## Task decomposition strategy

The slice-level RRI is **100 (Very high)** when scored as one task (auth + audit +
migration + new public delivery surface), which mandates decomposition before any
implementation (ADR-032 anticipates this; RRI policy ┬¦decomposition). The slice is
split so the grant contract, the pure rewriter, the schema/repo, the delivery wiring,
and authorization/observability can each be approved and verified independently:

1. `S-125-T0` ŌĆö author this plan + task ledger, BDD feature, and roadmap/ADR sync
   (planning/docs; no code).
2. `S-125-T1` ŌĆö playback-grant domain contract + fail-closed state decoding
   (`crates/domain`).
3. `S-125-T2` ŌĆö playback-grant schema + repository, resolving a grant to prepared-HLS
   lineage and readiness (`infra/migrations`, `crates/db`).
4. `S-125-T3b` ŌĆö scaffold the `crates/playback` crate + add the `crates/storage`
   scoped-reference helper (the no-raw-key boundary); retained by the primary agent.
5. `S-125-T3a` ŌĆö the pure, IO-free `rewrite_manifest` function + its unit tests, filled
   into the scaffolded crate. This is the **only** S-125 sub-part that scores RRI Ōēż 25
   (measured 21), so it is delegated to local Gemma via Ollama; the primary agent stays
   orchestrator/reviewer of record (see the ledger's "T3 split" note and
   `LOW_RRI_LOCAL_MODEL_HANDOFF.md`).
6. `S-125-T4` ŌĆö grant issuance API behind the verified principal + `S-100`/`S-160`
   review authorization, with durable grant audit (`apps/api`, observability). This
   is now further split into `T4a-i`, `T4a-ii`, `T4b-i`, `T4b-ii`, `T4b-iii`, `T4c`,
   and `T4d`.
7. `S-125-T5a` ŌĆö manifest delivery handler + rewritten-manifest integration tests.
8. `S-125-T5b` ŌĆö short-lived segment-reference generation, storage helper(s), and
   delivery observability (no durable per-segment audit).
9. `S-125-T5c` ŌĆö ADR-032 `Accepted` propagation + architecture/roadmap/BDD sync.

The original T3 was split because exactly one S-125 sub-part qualifies for local Gemma
delegation: the pure manifest rewriter clears RRI Ōēż 25 with legitimately low D/K/P,
while everything else is floored above the band by the anchor rubric (migrations ŌåÆ
D4/P5/K4; authz/audit ŌåÆ security-critical). T3b precedes T3a because Gemma fills only
the function body in an already-scaffolded crate.

No implementation should start outside those bounded tasks, and each code task carries
its own RRI, Reflection strategy, and approval checkpoint in the task ledger.

## Relationship to adjacent slices

- **Built on:** `S-120` prepared HLS package + `PreparationStatus` readiness;
  `S-080` production-like `StorageAdapter`; `S-000`/`S-023` verified principal;
  `S-100`/`S-160` org/project authorization model (ADR-027, ADR-030).
- **Unblocks directly:** `S-170` human review runtime (review preview playback) and
  `S-180` publication runtime (audience-facing playback gated by ADR-030).
- **Does not solve:** the HLS package contents (`S-120`), audience visibility policy
  and CDN/public distribution (`S-180` + future public-player decision, ADR-029),
  DRM/encryption, or alternate rendition ladders.
